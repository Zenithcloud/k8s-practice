Kubernetes RBAC 

👤 Users and Groups
Users:

Represent real people (cluster administrators, developers, operators).

Kubernetes itself does not store user accounts — authentication is handled externally (certificates, OIDC, LDAP, etc.).

A user identity is passed to the API server when they log in.

Example: alex or sam logging in with a client certificate.

Groups:

Logical collections of users.

Useful for assigning permissions to multiple people at once.

Example: developers group can be given read access to pods in a namespace.

Utilization:

Used for human access control.

You bind Roles/ClusterRoles to users or groups so that people can perform actions like deploying apps, viewing logs, or managing resources.

🛠 Service Accounts
Service Accounts:

Represent non-human identities (used by applications, controllers, or pods).

Exist as Kubernetes objects (kind: ServiceAccount) inside a namespace.

Automatically created for each namespace (e.g., default service account).

Pods can be configured to run under a specific service account, and they use its credentials to talk to the API server.

Example: A monitoring pod uses a service account with permission to read metrics from the cluster.

Utilization:

Used for workload access control.

Bind Roles/ClusterRoles to service accounts so that applications can securely interact with cluster resources.

Example: A CI/CD pipeline pod needs permission to create deployments in dev namespace.

📊 Comparison Table
Aspect	Users/Groups (Human)	Service Accounts (Non-Human)
Identity Source	External auth (certs, OIDC, LDAP)	Kubernetes object in a namespace
Scope	Cluster-wide	Namespace-scoped
Typical Use	Developers, admins, operators	Pods, controllers, automated processes
Management	Outside Kubernetes (kubeconfig, IAM)	Inside Kubernetes (ServiceAccount YAML)
Binding	RoleBinding / ClusterRoleBinding	RoleBinding / ClusterRoleBinding
📝 Summary
Users/Groups → for people accessing the cluster.

Service Accounts → for applications or pods running inside the cluster.

Both are tied to RBAC via RoleBindings or ClusterRoleBindings, but they serve different audiences: humans vs workloads.
#######################################################################################################################

RBAC (Role-Based Access Control) lets you control what users can do in your cluster.

Role
A Role gives permissions inside one namespace.

yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
Apply it:

bash
kubectl apply -f role.yaml
kubectl get role
RoleBinding
A RoleBinding connects a Role to a user in one namespace.

yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: default
subjects:
- kind: User
  name: alex
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
Apply it:

bash
kubectl apply -f rolebinding.yaml
kubectl get rolebinding
kubectl auth can-i get pod --as alex
ClusterRole
A ClusterRole gives permissions across the whole cluster.

yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: secret-reader
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "watch", "list"]
Apply it:

bash
kubectl apply -f clusterrole.yaml
kubectl get clusterrole
RoleBinding (Namespace-level with ClusterRole)
You can bind a ClusterRole to a user in one namespace.

yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-secrets
  namespace: development
subjects:
- kind: User
  name: sam
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
Apply it:

bash
kubectl apply -f rolebinding.yaml
kubectl get rolebinding
kubectl auth can-i get secret --as sam -n development
ClusterRoleBinding
A ClusterRoleBinding connects a ClusterRole to a user across all namespaces.

yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: read-secrets-global
subjects:
- kind: User
  name: maya
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
Apply it:

bash
kubectl apply -f clusterrolebinding.yaml
kubectl get clusterrolebinding
kubectl auth can-i get secret --as maya -A
📝 Summary
Role → permissions in one namespace.

RoleBinding → connects Role to user in one namespace.

ClusterRole → permissions across all namespaces.

ClusterRoleBinding → connects ClusterRole to user across all namespaces.

🔧 Steps to Create a Certificate User (alex)
1. Generate a Private Key
bash
openssl genrsa -out alex.key 2048
2. Create a Certificate Signing Request (CSR)
bash
openssl req -new -key alex.key -out alex.csr -subj "/CN=alex/O=devteam"
CN=alex → Common Name, which becomes the Kubernetes username.

O=devteam → Organization, which becomes the Kubernetes group (optional).

3. Sign the Certificate with the Cluster CA
You need access to your cluster’s CA key and certificate (usually found in /etc/kubernetes/pki/ on the control plane).

bash
openssl x509 -req -in alex.csr -CA /etc/kubernetes/pki/ca.crt \
-CAkey /etc/kubernetes/pki/ca.key -CAcreateserial -out alex.crt -days 365
Now you have:

alex.key → private key

alex.crt → signed certificate

4. Add User to kubeconfig
Create a new user entry in your kubeconfig:

bash
kubectl config set-credentials alex \
  --client-certificate=alex.crt \
  --client-key=alex.key
5. Create a Context for Alex
bash
kubectl config set-context alex-context \
  --cluster=kubernetes \
  --namespace=default \
  --user=alex
Switch to that context:

bash
kubectl config use-context alex-context
6. Test RBAC
Now, when you run:

bash
kubectl auth can-i get pods
It will check RBAC for the user alex.
If you applied the Role + RoleBinding earlier, you should see:

Code
yes
📝 Summary
Certificates let you create real Kubernetes users.

The CN field in the certificate must match the username in your RoleBinding (alex).

Once added to kubeconfig, you can switch contexts and test RBAC rules directly.
